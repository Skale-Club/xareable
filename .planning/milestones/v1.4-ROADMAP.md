## Phase Details

### Phase 17: GHL Signup Sync (Wire-Up)

**Goal:** When a new Xareable user signs up AND the admin has opted in, the user lands inside the operator's connected GHL location as a contact tagged `xareable` — without ever blocking, slowing, or breaking the signup flow itself, and with every push (success or failure) visible to ops in the existing integration delivery log.

**Depends on:** Phase 16 (v1.3 closed; v1.4 starts on a clean main). Code-level: depends on the existing GHL admin surface (`server/integrations/ghl.ts`, `PATCH /api/admin/ghl`, the GHL card in `integrations-tab.tsx`), the existing signup hook (`POST /api/telegram/notify-signup` at `server/routes/integrations.routes.ts:1863`), the existing `integration_delivery_logs` table (used today for telegram signup deliveries), and the existing `integration_settings.ghl` row.

**Requirements:** GHL-01, GHL-02, GHL-03

**Success Criteria** (what must be TRUE when this phase ships):

1. **Server push fires on signup, opt-in gated.** When a new user signs up and the existing signup hook (`POST /api/telegram/notify-signup` — the same route that already fires `trackMarketingEvent({ event_name: "CompleteRegistration", event_key: "signup:<user.id>", … })`) runs, AND the GHL integration row in `integration_settings` has `enabled=true` AND the new opt-in flag (`sync_on_signup`) is true, the server invokes `getOrCreateGHLContact()` from `server/integrations/ghl.ts` with `email` from the auth user, `firstName`/`lastName` parsed from `user.user_metadata` when present (with safe fallback when absent), and `tags: ['xareable']`. When either `enabled=false` or `sync_on_signup=false`, the call is skipped and nothing is sent to GHL. Idempotency on repeat signup-event delivery is preserved by the existing `event_key='signup:<user.id>'` uniqueness pattern (mirrors how telegram delivery is deduped today). [GHL-01]

2. **Admin opt-in checkbox visible, persisted, immediately reflected.** The GHL card at `client/src/components/admin/integrations-tab.tsx:1054` (the GoHighLevel card with the existing `saveGhlMutation` PATCHing `/api/admin/ghl`) gains a checkbox labeled exactly **"Sync new signups to GHL (tagged `xareable`)"** with the help text **"When enabled, every new Xareable user is automatically created as a contact in your connected GoHighLevel location, tagged `xareable`. Use this tag to trigger campaigns or workflows inside GHL."** The checkbox defaults to `false`. Saving the GHL settings persists the value via the existing PATCH endpoint, the GET response surfaces it back, and the checkbox reflects the saved state on next render WITHOUT a page reload (existing `queryClient.invalidateQueries` on the GHL query key already handles this — the new field rides the same invalidation). [GHL-02]

3. **Best-effort: signup never blocked, never raises a user-visible error.** If `getOrCreateGHLContact()` throws OR the GHL API returns a non-2xx status (handled inside the wrapper which already returns `{ success: false, error }` instead of throwing — see `server/integrations/ghl.ts:382`), the signup hook completes successfully (HTTP 200 to the client) regardless. A row is written to `integration_delivery_logs` (the same table already used by the telegram signup branch — rows scoped by `integration_type='ghl'`, `event_name='CompleteRegistration'`, `event_key='signup:<user.id>'`, `user_id=<user.id>`, `status='sent'|'failed'|'skipped'`, `reason` populated on failure/skip, `payload` carrying `{ contact_id, created }` on success). On success the GHL contact ID is observable in `payload.contact_id`. On failure, the error message is observable in `reason`. Server logs `[GHL] sync ok|fail user=<id> reason=...` with the same prefix convention used by the existing GHL log lines (`server/routes/integrations.routes.ts:359` etc). [GHL-03]

4. **Test panel + manual verification confirms end-to-end push.** With a fresh test GHL location configured (admin enables GHL, sets API key + Location ID, ticks "Sync new signups to GHL", saves), creating a new test Xareable user via the standard signup flow results in (a) a new contact appearing in the GHL location with the expected email + tag `xareable`, (b) one `integration_delivery_logs` row with `integration_type='ghl'`, `status='sent'`, `payload.contact_id` set to the GHL contact ID, and (c) NO regression in the existing telegram signup branch (its delivery log row continues to be written as before).

5. **No regressions.** `npm run check` and `npm run build` both pass. The existing GHL admin GET/PATCH/test/custom-fields endpoints continue to behave identically when `sync_on_signup` is absent/false. The existing `trackMarketingEvent` `CompleteRegistration` flow continues to fire (GA4 + Facebook delivery branches are untouched). Toggling `sync_on_signup` off (via PATCH) cleanly disables the new branch — confirmed by a fresh signup recording a `status='skipped'` delivery log with `reason='sync_on_signup_disabled'` (or equivalent reason text — exact tokens are a planner decision).

**Planning Concerns** (to resolve during `/gsd:plan-phase 17`):

> The original requirement document assumed several specific storage shapes that don't exist in the codebase as written. None of the deltas blocks the requirements — they are storage-shape decisions for the planner. Specifically:

- **a. Signup event name is `CompleteRegistration`, not `signup`.** The existing signup hook (`POST /api/telegram/notify-signup` at `integrations.routes.ts:1863`) calls `trackMarketingEvent({ event_name: "CompleteRegistration", event_key: "signup:<user.id>", event_source: "auth", … })`. The seed/PROJECT.md text says "`event_type='signup'`"; in code the `event_name` is `CompleteRegistration` and the `event_key` is `signup:<user.id>`. The GHL push branch should hook into the same handler at the same point (immediately after the `trackMarketingEvent` invocation around line 1901, in parallel with the telegram branch that follows).

- **b. `marketing_events.delivery_status` JSONB does NOT exist.** The `marketing_events` table (migration `20260305220000_marketing_events.sql`) carries fixed columns `ga4_status` / `ga4_response` / `facebook_status` / `facebook_response` — no general-purpose `delivery_status` JSONB. **Recommended path (matches existing pattern):** record GHL delivery in `integration_delivery_logs` (migration `20260307000000_integration_observability.sql`), the table already used by the telegram signup branch via `recordIntegrationDeliveryLog()`. Schema already supports it: `integration_type text` (use `'ghl'`), `event_name text` (use `'CompleteRegistration'`), `event_key text` (use `'signup:<user.id>'`), `status text` (`sent|failed|skipped|queued`), `reason text`, `user_id uuid`, `payload jsonb`. **Alternative paths:** (b1) extend `marketing_events` with a `delivery_status jsonb` column (more invasive, cross-cuts GA4/FB shape); (b2) add a small `marketing_events_ghl` join table. Option (a) — reuse `integration_delivery_logs` — is the natural cut and keeps GHL/telegram parallel.

- **c. `integration_settings.ghl.settings` JSONB does NOT exist.** The `integration_settings` table (migration `20260305000013_integration_settings.sql`) has fixed columns: `enabled`, `api_key`, `location_id`, `custom_field_mappings jsonb`, `last_sync_at`, `enabled_at`. There is no general-purpose `settings` JSONB. **Storage options for `sync_on_signup`:** (c1) add a small additive migration with a `sync_on_signup boolean DEFAULT false` column on `integration_settings` (cleanest typing; widens the table for one boolean — but `integration_settings` is a small admin-only table so widening is cheap); (c2) stash it inside `custom_field_mappings` JSONB under a synthetic reserved key (e.g. `__sync_on_signup`) — zero migration but conflates "field mappings" with "feature flags"; (c3) add a generic `settings jsonb DEFAULT '{}'` column once and use it now plus future flags. Option (c1) is the most type-safe; option (c3) is the most future-flexible. Either is in v1.4 scope. Choice is a planning decision; mirror it in the Zod schema (`saveGHLSettingsRequestSchema`, `adminGHLStatusSchema`) accordingly.

- **d. The `getOrCreateGHLContact` wrapper does NOT today thread `tags` through to the create path.** The signature at `server/integrations/ghl.ts:382` accepts `GHLContactPayload` which DOES include `tags: z.array(z.string()).optional()` per `shared/schema.ts:605`, and `createGHLContact` spreads the full payload into the request body — so tags WILL be sent if passed. ✓ No change needed in the wrapper. (The planner should verify by inspecting the request body shape to GHL `POST /contacts/`.)

- **e. The signup hook (`POST /api/telegram/notify-signup`) is a slightly mis-named home for the GHL push.** Today it does both signup-event tracking AND telegram notification. Adding GHL alongside makes the route name lie a little. **Options:** (e1) keep the route as-is and add the GHL branch in parallel with the telegram branch (zero churn, mild naming smell); (e2) rename to `POST /api/auth/notify-signup` or `POST /api/integrations/notify-signup` (cleaner naming, but a client-side caller change in `client/src/lib/auth.tsx` and a deprecation alias on the old path). v1.4 bias: option (e1) — keep the route, accept the naming smell, document in a code comment. Refactoring routes is out of scope for v1.4.
